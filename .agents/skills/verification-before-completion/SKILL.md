# 完成前驗證

## 概述

在沒有驗證的情況下宣稱工作完成，這不是高效，而是不誠實。

**核心原則：** 始終用證據支撐結論。

**對這條規則敷衍了事，就等於違背了它的精神。**

## 鐵律

```
沒有新鮮的驗證證據，不許宣稱完成
```

如果你在這條訊息中沒有執行驗證命令，就不能聲稱測試通過。

## 門控函式

```
在宣稱任何狀態或表達滿意之前：

1. 確定：什麼命令能證明這個結論？
2. 執行：執行完整命令（重新執行，完整執行）
3. 閱讀：完整輸出，檢查退出碼，統計失敗數
4. 驗證：輸出是否支援這個結論？
   - 如果否：用證據說明實際狀態
   - 如果是：帶證據陳述結論
5. 只有這時：才能做出結論

跳過任何一步 = 說謊，不是驗證
```

## 常見失敗模式

| 結論 | 需要 | 不夠格 |
|------|------|--------|
| 測試通過 | 測試命令輸出：0 failures | 之前的執行結果、"應該會通過" |
| Linter 無報錯 | Linter 輸出：0 errors | 部分檢查、推斷 |
| 構建成功 | 構建命令：exit 0 | linter 通過、日誌看起來沒問題 |
| Bug 已修復 | 測試原始症狀：通過 | 程式碼改了，假設已修復 |
| 迴歸測試有效 | 紅-綠迴圈已驗證 | 測試只通過了一次 |
| 代理已完成 | VCS diff 顯示變更 | 代理報告"成功" |
| 需求已滿足 | 逐項核對清單 | 測試通過 |

## 紅線——停下來

- 使用"應該"、"大概"、"似乎"
- 驗證前就表達滿意（"太好了！"、"完美！"、"搞定！"等）
- 即將提交/推送/建立 PR 卻沒有驗證
- 信任代理的成功報告
- 相依性部分驗證
- 想著"就這一次"
- 累了想趕緊收工
- **任何暗示成功但實際未執行驗證的措辭**

## 防止合理化

| 藉口 | 現實 |
|------|------|
| "應該能行了" | 執行驗證命令 |
| "我有信心" | 信心 ≠ 證據 |
| "就這一次" | 沒有例外 |
| "Linter 通過了" | Linter ≠ 編譯器 |
| "代理說成功了" | 獨立驗證 |
| "我累了" | 疲勞 ≠ 藉口 |
| "部分檢查就夠了" | 部分檢查什麼也證明不了 |
| "換個說法這條規則就不適用了" | 精神大於字面 |

## 關鍵模式

**測試：**
```
✅ [執行測試命令] [看到：34/34 pass] "全部測試通過"
❌ "應該能通過了" / "看起來對了"
```

**迴歸測試（TDD 紅-綠）：**
```
✅ 編寫 → 執行（通過）→ 回退修復 → 執行（必須失敗）→ 恢復 → 執行（通過）
❌ "我寫了迴歸測試"（沒有經過紅-綠驗證）
```

**構建：**
```
✅ [執行構建] [看到：exit 0] "構建通過"
❌ "Linter 通過了"（linter 不檢查編譯）
```

**需求：**
```
✅ 重讀計劃 → 建立核對清單 → 逐項驗證 → 報告缺口或完成
❌ "測試通過了，階段完成"
```

**代理委派：**
```
✅ 代理報告成功 → 檢查 VCS diff → 驗證變更 → 報告實際狀態
❌ 信任代理報告
```

## 為什麼這很重要

來自 24 次失敗記錄：
- 搭檔說"我不信你"——信任被破壞
- 未定義的函式被交付——會直接崩潰
- 遺漏需求被交付——功能不完整
- 虛假完成浪費的時間 → 返工 → 重做
- 違反原則："誠實是核心價值。如果你說謊，就會被替換。"

## 何時使用

**以下情況之前必須使用：**
- 任何形式的成功/完成宣告
- 任何滿意的表達
- 任何關於工作狀態的正面陳述
- 提交、建立 PR、標記任務完成
- 進入下一個任務
- 委派給代理

**本規則適用於：**
- 準確措辭
- 同義詞和換一種說法
- 暗示成功
- 任何傳達完成/正確性的溝通

## 底線

**驗證沒有捷徑。**

執行命令。閱讀輸出。然後才能宣稱結果。

這沒有商量餘地。
