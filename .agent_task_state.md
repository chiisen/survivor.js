## 學到的經驗
- **Pi vs Claude Code 風格差異**: Pi 在 void 方法選擇加 `@returns {void}` (utils.js 是省略),展現「主動表達」傾向;Claude Code 偏向「最小變更」嚴格對齊標竿。兩者都正確,只是品味不同。
- **Pi 適合的任務類型**: 機械化、範圍明確 (單檔 JSDoc 補註) 適合;需要技能紀律或跨檔重構不適合。
- **監督者必須自跑驗證**: 不能信任 Pi 的自我報告 (Pi 可能會說「測試通過」但實際沒跑或跑了不同指令),`npm test` 與 `git diff` 必須由 Claude Code 自己執行。
- **Windows + Pi 寫檔**: 可能觸發 LF/CRLF 警告 (非錯誤),不影響功能。
- **AGENTS.md 是軟性規範,`--append-system-prompt` 才是強制 (2026-07-11 實測)**: 只寫 AGENTS.md 要求 Pi echo 進度,Pi 完全不遵守 (2 分鐘無產出);改用 `--append-system-prompt` 強制後,Pi 完美執行 4 步驟證 (pi-progress.log 90 bytes 齊全)。**省 token 進度回報機制必須靠 system prompt 強制,不能靠 AGENTS.md 軟性規範**。
- **Pi 處理 JSDoc 補註任務不穩定 (5 次測試僅 1 次完全成功)**: v3 objectPool.js (169 行) 中止時剛好完成;v4 talent.js (@typedef 進階) timeout 130s 完全沒產出;v5 talent.js (簡化 prompt) timeout 200s 完全沒產出。**簡化 prompt 與延長 timeout 都無效,根因在 minimax-m3 model 端**。結論:正式 JSDoc 工程建議由 Claude Code 自己做,Pi 派發保留給「實驗性 / 重複性高 / 範圍極明確」的任務。
